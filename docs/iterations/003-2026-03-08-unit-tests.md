# Iteration 003 — Unit tests and CLI testability

**Date:** 2026-03-08
**Branch:** `feat/unit-tests`

## What was done

### Problem being solved

Haven had ~1200 lines of code and only 2 test files (`certutil`, `bootstrap`) — roughly 10% coverage. CLI commands were untestable because they called `buildProviderAndStore()` internally, coupling them to real AWS credentials.

### Solution

Two-part approach:
1. Write unit tests for all pure functions across the codebase
2. Refactor CLI commands with dependency injection to enable mock-based testing

### CLI dependency injection refactor

Changed `runDeploy`, `runDestroy`, `runStatus`, `runCert` signatures to accept `provider.Provider` and `provider.StateStore` as parameters. Cobra `RunE` wrappers call `buildProviderAndStore` and pass results down. No behavior changes.

Before:
```go
func runDestroy(ctx, providerName, deploymentID, verbose) error {
    prov, store, err := buildProviderAndStore(ctx, providerName, out)
    // ...
}
```

After:
```go
func runDestroy(ctx, prov, store, deploymentID, verbose) error {
    // prov and store injected — testable with mocks
}
```

## What works

`go build ./...` passes. `go test ./...` passes — 32 tests across 8 packages.

Test coverage highlights:
- All pure functions covered (model registry, CFN terminal checks, template generation, key/ID generation)
- CLI error paths covered via mocks (not-found, API failures, missing data)
- Mock infrastructure ready for future tests

## What's not covered

- `runDeploy` happy path (requires mocking `detectPublicIP`, `waitForOllama`, file I/O)
- AWS SDK integration (S3 CRUD, CloudFormation calls) — would need SDK mocks or LocalStack
- `pollStackEvents` loop logic
- TUI spinner (goroutine-based, terminal output)
